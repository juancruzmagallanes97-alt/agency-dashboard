# Plan 01-05 — Settings / Config Screen

## Context

Phase 1 success criteria #6: admin reaches Settings, sees connection status for every
integration, and the platform shows live status per tool.

This plan delivers CONF-01 (config storage and reading) and CONF-02 (live status UI).
Config values are read from server-side environment variables — the browser never
touches credentials directly.

## Tasks

### Task 1 — apps/dashboard/lib/config.ts

Create a typed config reader with a ConnectionConfig interface (n8n, airtable, ghl,
openai, slack). Export a getConfig() function that reads from process.env at request
time. Export a ToolName union type.

Env var names to read:
- n8n: N8N_API_URL, N8N_API_KEY
- airtable: AIRTABLE_API_KEY, AIRTABLE_BASE_ID
- ghl: GHL_API_KEY
- openai: OPENAI_KEY
- slack: SLACK_WEBHOOK_URL

Empty string default (not undefined) for all missing vars so downstream code can
do simple truthiness checks.

### Task 2 — apps/dashboard/app/api/test-connection/route.ts

Server-side GET handler. Accepts query param tool=n8n|airtable|ghl|openai|slack.
Calls getConfig(), probes the integration, returns { ok: boolean, latencyMs: number, error?: string }.

Probe strategy per tool:
- n8n: GET {url}/api/v1/workflows with Authorization: Bearer {apiKey} — expect 200
- airtable: GET https://api.airtable.com/v0/meta/bases with Bearer — expect 200
- openai: GET https://api.openai.com/v1/models with Bearer — expect 200
- slack: POST {webhookUrl} with empty JSON body — expect 200
- ghl: config-present check only (apiKey non-empty = ok, no public health endpoint)

Returns 400 for unknown tool. On network error, sets ok=false and error=message.

### Task 3 — apps/dashboard/app/configuracion/page.tsx

Client Component. One status card per integration showing:
- Tool name, icon/emoji, configured URL or masked credential hint
- Status dot: green (connected) / red (error) / gray (unconfigured / not tested yet)
- Latency badge when ok
- "Testear" button that calls /api/test-connection?tool=X and updates the card state
- "Testear todas" button that fires all 5 calls in parallel via Promise.all

No credential editing form in this plan — that is CONF-02 extended scope, deferred.
Showing live status satisfies Phase 1 criterion #6.

### Task 4 — Sidebar wiring

Edit apps/dashboard/components/Sidebar.tsx. Add a "Configuración" nav item pointing
to /configuracion. Place it in the bottom utility section near any existing links.

### Task 5 — Build verification

Run from apps/dashboard/: ./node_modules/.bin/next build
Must exit 0 with 0 TypeScript errors and 0 ESLint errors.

## Acceptance Criteria

- [ ] /configuracion renders without errors
- [ ] Each of the 5 integration cards shows an initial status indicator
- [ ] Clicking "Testear" fires the server proxy and updates that card's UI
- [ ] "Testear todas" fires all 5 in parallel and updates all cards
- [ ] No credential appears in any browser network request
- [ ] next build passes clean

## Notes

- Credentials are stored in the dashboard environment file (gitignored). Unconfigured
  tools show gray "Sin configurar" — not an error state.
- n8n base URL defaults to the known server IP if the env var is absent (fallback
  lives in the API route, not in config.ts).
