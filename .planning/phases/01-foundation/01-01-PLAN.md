**Paso 2 — Crear apps/dashboard/auth.ts** (implementación completa, no stub):
```typescript
import NextAuth from "next-auth"
import Credentials from "next-auth/providers/credentials"
import bcryptjs from "bcryptjs"

export const { handlers, signIn, signOut, auth } = NextAuth({
  providers: [
    Credentials({
      credentials: {
        email: {},
        password: {},
      },
      authorize: async (credentials) => {
        const email = credentials.email as string
        const password = credentials.password as string

        if (email !== process.env.ADMIN_EMAIL) return null

        const hash = process.env.ADMIN_PASSWORD_HASH
        if (!hash) return null

        const valid = await bcryptjs.compare(password, hash)
        if (!valid) return null

        return { id: '1', email, name: 'Admin' }
      },
    }),
  ],
  session: { strategy: 'jwt' },
  pages: { signIn: '/login' },
  callbacks: {
    authorized: async ({ auth }) => !!auth,
  },
})
```
NOTA: sin console.log, sin semicolons al final (seguir estilo del proyecto).

**Paso 3 — Crear apps/dashboard/proxy.ts** (con lógica custom de redirect):

CRÍTICO: Este archivo se llama `proxy.ts`, NO `middleware.ts`. En Next.js 16, el archivo convencional es `proxy.ts` con export `proxy`. Crear `middleware.ts` es incorrecto.

```typescript
import { auth } from "@/auth"

export const proxy = auth((req) => {
  const isLoggedIn = !!req.auth
  const isOnLogin = req.nextUrl.pathname === '/login'

  if (!isLoggedIn && !isOnLogin) {
    return Response.redirect(new URL('/login', req.nextUrl.origin))
  }
  if (isLoggedIn && isOnLogin) {
    return Response.redirect(new URL('/', req.nextUrl.origin))
  }
})

export const config = {
  matcher: ['/((?!_next/static|_next/image|favicon.ico|api/auth).*)'],
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → proxy.ts | Todas las solicitudes HTTP entran por aquí antes de llegar a App Router |
| proxy.ts → auth.ts | proxy.ts llama auth() para verificar el JWT cookie |
| auth.ts → process.env | authorize() lee ADMIN_EMAIL y ADMIN_PASSWORD_HASH desde env vars |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | auth.ts — authorize() | mitigate | bcryptjs.compare() es time-constant por diseño del algoritmo bcrypt — previene timing attacks en comparación de password |
| T-01-02 | Information Disclosure | auth.ts — credenciales | mitigate | ADMIN_EMAIL y ADMIN_PASSWORD_HASH solo en .env.local (gitignored) — nunca en código fuente ni historial git |
| T-01-03 | Elevation of Privilege | proxy.ts — route protection | mitigate | matcher cubre todas las rutas excepto _next/static, _next/image, favicon.ico y api/auth — no hay rutas sin protección accidental |
| T-01-04 | Tampering | proxy.ts — JWT cookie | mitigate | next-auth v5 usa HttpOnly + Secure + SameSite=lax cookies por defecto — JavaScript del browser no puede leer ni modificar el token |
| T-01-05 | Information Disclosure | globals.css / layout.tsx | accept | CSS tokens y layout son públicos por naturaleza — no contienen secretos, riesgo inexistente |
</threat_model>
