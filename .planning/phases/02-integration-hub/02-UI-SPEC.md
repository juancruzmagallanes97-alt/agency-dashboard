### Skeleton loading state (per card)

While `metricsLoaded === false` (Promise.all pending), each card shows a skeleton in place of the metric line:

```

```

- Width: 96px (w-24), height: 12px (h-3)
- Background: `var(--border)` at 60% opacity
- Animation: CSS `opacity` pulsing 1.0 → 0.4 → 1.0 over 1.2s (use Tailwind `animate-pulse`)
- Position: exactly where the metric text would appear (same line, same left margin)
- No skeleton for the tool name, description, status dot, or test button — only the metric line

### ChecklistItem spec

```
div: flex items-start gap-3 py-1

  [checkbox area]
    button: w-4 h-4 rounded border border-[--border-2] shrink-0
    checked state: bg-[--accent] border-[--accent] — filled accent square with white checkmark ✓
    unchecked state: bg-transparent border-[--border-2]

  [text area]
    label: 13px text-2 (checked: text-3 + line-through)
    sub (tool name): 11px text-3 mt-2 if envVar provided: "Var: {envVar}" in mono

  Interaction: clicking toggles checked state via local useState
```

**Interaction model decision:** Inline accordion (not modal, not side panel). The detail panel expands as a sibling element immediately below the clicked item, pushing content down. One panel open at a time — clicking a second item collapses the first. Clicking the active item again collapses it.

Rationale: The page is a single-column narrow content area. A side panel would require layout restructuring incompatible with the existing shell. A modal breaks the "reference while scrolling" pattern needed to cross-check tool config. Accordion is consistent with Notion's inline expand pattern.

## Interaction Model

### Integration cards — metrics load sequence

1. Component mounts (`useEffect` with `[]` dependency)
2. `metricsLoading = true` → all 6 cards show skeleton line
3. `Promise.all([fetchN8nMetric(), fetchChatwootMetric(), ...])` fires in parallel
4. As each promise resolves, update state per tool: `metrics[tool] = result`
5. When all settle (via `Promise.allSettled` to avoid one failure blocking others), `metricsLoading = false`
6. Cards render metric text; failed metrics show "—" (em dash, no error message)

**State shape:**
```typescript
type MetricState = { loading: true } | { loading: false; value: string }
// value is already-formatted string: "12 workflows activos", "Webhook activo", etc.
// "—" for fetch error (silent — do not surface error to user for metrics)
```

### Individual test button — behavior (unchanged from Phase 1)

Click "Testear" → StatusDot goes amber → API call → result shows in StatusDot. No change to existing behavior. Metrics and status are independent states.

### Automation catalog — interaction sequence

1. Admin sees list of CatalogItems (all collapsed)
2. Click item → `selectedId` setState updates → that item expands inline
3. Click same item → `selectedId = null` → collapses
4. Click different item → first collapses, second expands (single-open accordion)
5. Checklist items maintain their own checked state via `Map<string, boolean>` in useState

**No persistence:** Checked items are in-memory only. On page reload, all items reset to unchecked. (Persistence deferred to Phase 3+.)

## Implementation Constraints (for executor)

1. **No new routes.** Everything in `app/configuracion/page.tsx`. Locked.
2. **No credential write path.** Read env vars only via `lib/config.ts`. Locked.
3. **Client component.** Page stays `'use client'` — metrics fetch requires `useEffect`. Existing pattern.
4. **Add Chatwoot to `lib/config.ts`:** `chatwoot: { apiUrl: string; apiToken: string; accountId: string }`. Add `'chatwoot'` to `ToolName` union.
5. **Metrics fetch lives in an API route.** `app/api/test-connection/route.ts` extended with `?tool=chatwoot` and a new `?type=metrics` mode (or separate `/api/metrics` route — planner decides). Credentials never flow to client.
6. **`Promise.allSettled`** (not `Promise.all`) for metrics — one failing fetch must not block all others.
7. **Tailwind v4 syntax.** Use `@import "tailwindcss"` pattern. No `tailwind.config.js`. Arbitrary values use bracket notation: `text-[11px]`, `text-[13px]`, `border-l-[3px]`.
8. **Inline `style` + Tailwind coexistence.** Follow existing pattern: layout classes in Tailwind, colors via `style={{ color: 'var(--token)' }}`.
9. **No `console.log`.** Silent error handling per codebase convention.
10. **Sub-components as `function` declarations** inside the page file, not exported. Follow `WfRow`, `TareaRow` pattern.
11. **`Promise.allSettled` result mapping:** index must match tool order. Use `Object.fromEntries` or explicit destructuring — do not rely on implicit order.
12. **Unicode icons are decorative.** All Unicode glyphs used as tool icons (e.g. ⚙ ⊞ ◈ ◉ ⊕ 💬 ◎) must carry `aria-hidden="true"` since the tool name is already present as adjacent visible text. Do not add `aria-label` to decorative icons.
